## 2. File Naming Convention

### 2.1 Primary Convention: Semantic Versioning

All project output files within the project root directory MUST use semantic versioned filenames:

```
MAJOR.MINOR[.PATCH].ext
```

| Component | Meaning | Example |
|:----------|:--------|:--------|
| **MAJOR** | Sequential project-wide iteration number. First output = 1, second = 2, thirteenth = 13. Increments for each distinct chat thread or project phase product. | `0.4.md` |
| **MINOR** | Sub-iteration within a major version. Increment when revising or extending a document without starting a new thread. | `0.2.1.md` |
| **PATCH** | Minor variant, fix, or alternative of the same sub-iteration. Use sparingly — for typo fixes or format corrections only. | `0.2.1.1.md` |

### 2.2 Associated File Rule

Files associated with a primary document (Python scripts, data files, images) MUST share the same version number as their parent document:

| Parent Document | Associated File | Relationship |
|:----------------|:----------------|:-------------|
| `0.4.md` (Literature Review) | `0.4_search.py` | Search script that produced data for the review |
| `0.5.md` (Research Plan) | `0.5_diagram.png` | Diagram referenced in the plan |

### 2.3 Prohibited Patterns

| Prohibited | Correct Alternative | Reason |
|:-----------|:--------------------|:-------|
| `introduction.md` | `0.1.md` | Descriptive names are meaningless in a flat directory |
| `analysis.py` | `0.3.py` | Must match parent document version |
| `figure1.png` | `0.4_fig.png` | Must match parent document version |
| `final_v2_revised.md` | `0.1.2.md` | Meaningless qualifiers |
| `0.1 (2).md` | Delete duplicate; keep `0.1.md` | Windows duplicate suffix — never create |
| `0.1.2md` | `0.1.2.md` | Missing dot in extension |

### 2.4 Exception: Imported Source Files

Source files imported from external searches (research papers, references, web results) may use a prefixed format to preserve identifiability:

```
src_<MAJOR.MINOR>_<shortref>.md
```

| Example | Explanation |
|:--------|:------------|
| `src_0.4_smith2023.md` | Smith 2023 paper, imported for literature review 0.4 |
| `src_0.4_witten1987.md` | Witten 1987 adelic string paper, imported for lit review |

The version prefix (`0.4`) ties the source to the project phase that required it.

### 2.5 Exception: Module Report Files

Module execution report files may use descriptive names to enable GitHub Pages rendering and human navigation:

```
module_XX_report.md
```

| Example | Explanation |
|:--------|:------------|
| `module_01_report.md` | Module 1 execution report |
| `module_03_report.md` | Module 3 execution report |

These files are the **final deliverables** and benefit from descriptive naming for repository visitors. However, all PRELIMINARY and WORKING documents use the strict versioned convention.

### 2.6 Image Files

Generated images use the module number as prefix:

```
images/XX_description.png
```

| Example | Explanation |
|:--------|:------------|
| `images/01_product_formula_verification.png` | Module 1, product formula verification plot |
| `images/03_landscape.png` | Module 3, adelic landscape heatmap |

## 3. Directory Organization

### 3.1 Flat Root Directory

ALL project source files, research documents, and working files reside in a single flat directory:

```
G:\My Drive\Obsidian\projects\Adelic Constraints on Quantum Field Theory\
```

**Rationale:** In a flat directory where every file belongs to the same project, descriptive filenames provide no organizational benefit and obscure the chronological relationship between files. Version numbers are the only meaningful namespace.

### 3.2 Subdirectories

Three subdirectories are permitted for organizational purposes:

| Subdirectory | Purpose | Contents |
|:-------------|:--------|:---------|
| `src/` | Python source code | All `.py` files: classes, functions, tests |
| `images/` | Generated figures | All `.png` files: plots, charts, diagrams |
| `data/` | Generated data | `.json`, `.csv` files: computed results, tables |

**Rule:** No markdown documents (`.md`) go in subdirectories. All `.md` files live in the project root. Subdirectories are for code, images, and data only.

### 3.3 File Count Management

The flat root directory should contain at most **~30 files** at any time (excluding subdirectory contents). If the count grows beyond this:

1. **Archive completed work:** Move superseded versions to `G:\My Drive\Archive\projects\<YYYY>\<MM>\Adelic Constraints on Quantum Field Theory\`
2. **Consolidate:** Merge minor variants into a single document
3. **Clean:** Remove temporary files (`.pyc`, test outputs, debug logs)

## 4. Version Control (Git)

### 4.1 Repository

The project is tracked in the Git repository at:

```
G:\My Drive\Obsidian\projects\
```

The Adelic Constraints on Quantum Field Theory project is a subdirectory within this repository.

### 4.2 Commit Message Standard

All commit messages MUST follow the **Conventional Commits** format:

```
<type>: <description>

[optional body]

[optional footer]
```

| Type | Use Case |
|:-----|:---------|
| `feat` | New module document, new Python script, new feature |
| `fix` | Bug fix in code, correction in document |
| `docs` | Documentation-only changes |
| `refactor` | Code restructuring without behavioral change |
| `test` | Adding or modifying tests |
| `chore` | Maintenance tasks (file cleanup, gitignore) |

**Commit message examples:**
```
feat: Add 0.4.md (Module 0.1: Literature Review)

Comprehensive review of 620 release files and LLM-inference literature.
Identified 14 high-relevance prior publications by Quni-Gudzinas.
Gap analysis and recommendations included.

Files: 0.4.md
```

```
fix: Correct product formula verification in src/padic.py

The norm() method was incorrectly computing p-adic valuation for
denominators containing p. Fixed by using Fraction.limit_denominator().

Files: src/padic.py, src/test_foundations.py
```

### 4.3 Commit Frequency

- **Commit after every completed module document** (e.g., after writing `0.4.md`)
- **Commit after every completed Python script** (e.g., after `src/padic.py` passes tests)
- **Commit after every significant debug fix**
- **DO NOT commit after every minor edit** — batch small fixes into a single commit
- **DO NOT commit generated files** (`images/*.png`, `data/*.json`) unless they are final deliverables

### 4.4 Branching Strategy

**Default branch:** `main`

**Feature branches:** Not used for this project. All work happens on `main` with frequent commits. The commit history itself serves as the development log.

**Rationale:** The project is executed by a single LLM agent in sequential chat sessions. There is no concurrent development. Branches add overhead without benefit.

### 4.5 .gitignore

The following patterns should be in `.gitignore`:

```
__pycache__/
*.pyc
*.pyo
.pytest_cache/
*.egg-info/
.env
*.log
.ipynb_checkpoints/
```

Note: `images/` and `data/` are NOT gitignored — final deliverables should be committed.

## 5. File Lifecycle

### 5.1 Creation

1. **Determine next version number** using Python: scan the project directory for existing versioned files
2. **Create the file** with the determined version number
3. **Commit immediately** with a descriptive message

### 5.2 Iteration

1. **For minor edits** (typo fixes, formatting): increment PATCH → `0.1.2.1.md`
2. **For substantive revisions** (new section, expanded content): increment MINOR → `0.1.3.md`
3. **For major new outputs** (new module, new phase): increment MAJOR → `0.2.md`
4. **Always commit** after each version increment

### 5.3 Archival

When a file is superseded by a newer version:

1. **Move** the old file to `G:\My Drive\Archive\projects\<YYYY>\<MM>\Adelic Constraints on Quantum Field Theory\`
2. **Commit** with message: `archive: Move 0.1.md to archive (superseded by 0.2.md)`
3. **Do NOT delete** — the archive preserves the full development history

### 5.4 Final State

At project completion, the root directory should contain:
- All `0.N.md` research documents (current versions only)
- All `module_XX_report.md` execution reports
- `README.md`, `synthesis.md`, `requirements.txt`, `_config.yml`
- `src/` with final Python code
- `images/` with final figures
- `data/` with final datasets

Superseded versions are in the archive. The root directory is clean and publication-ready.

## 6. Provenance Tracking

### 6.1 Document-to-Document Traceability

Every document MUST include a metadata section at the bottom specifying:

```markdown
## Document Metadata

- **File:** `0.4.md`
- **Module:** 0.1 (Literature Review)
- **Based on:** (list of source files)
- **Supersedes:** (list of files this document replaces)
- **Superseded by:** (to be filled when a newer version exists)
- **Git commit:** (hash of the commit that created this version)
```

### 6.2 Code-to-Document Traceability

Every Python script MUST include a header comment:

```python
#!/usr/bin/env python3
"""
Module 1: Foundational Library
File: src/padic.py
Associated document: 0.5.md (Research Plan, Module 1 specification)
Report: module_01_report.md
Author: LLM agent (DeepSeek V4)
Date: 2026-05-09
"""
```

### 6.3 Image-to-Code Traceability

Every generated image filename encodes its origin:
- `XX_` prefix → Module number
- Descriptive suffix → What the image shows

The generating script is always `src/<script_name>.py` referenced in the associated module report.

## 7. Cleanup and Deduplication

### 7.1 Prohibited Duplicates

- **Never** create `filename (2).md` or `filename - Copy.md` — these are Windows artifacts
- **Never** create multiple files with the same version number
- **Never** create a file with identical content to an existing file

### 7.2 Detection

Before creating any file, run:

```python
import os
existing = os.listdir(project_dir)
if proposed_filename in existing:
    # INCREMENT VERSION NUMBER, DO NOT OVERWRITE
```

### 7.3 Regular Cleanup

After every 3 modules, run a cleanup pass:
1. Check for zero-byte files → delete
2. Check for duplicate content → keep higher version, archive lower
3. Check for `.pyc` files → delete (they're in `.gitignore`)
4. Verify no descriptive filenames in root directory

## 8. Restricted Operations

### 8.1 NEVER

| Operation | Reason |
|:----------|:-------|
| Use descriptive filenames (e.g., `introduction.md`) in root directory | Violates naming convention; meaningless in flat directory |
| Hardcode file paths in Python scripts | Use `os.path` or `pathlib` for portability |
| Delete any versioned file | Archive instead |
| Create files outside the project directory | All project files belong in `G:\My Drive\Obsidian\projects\Adelic Constraints on Quantum Field Theory\` |
| Modify files in `G:\My Drive\Archive\` | Archive is read-only historical record |
| Commit generated `.png` or `.json` files during development | Only commit final deliverable images/data |
| Use `pandas` in Python scripts | Standard library + `numpy/scipy/sympy/mpmath` only |

### 8.2 ALWAYS

| Practice | Reason |
|:---------|:-------|
| Use Python to determine next version number before creating files | Avoid collisions |
| Commit after every meaningful unit of work | Maintain granular history |
| Include metadata section in every document | Enable provenance tracking |
| Use `os.path.exists()` before writing files | Avoid overwrites |
| Store search scripts alongside their output | Reproducibility |
