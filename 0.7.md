## 3. Code Execution Protocol

### 3.1 Script Requirements

Every Python script in `src/` must satisfy:

1. **Self-contained:** Runnable with `python script_name.py` from the project root
2. **No hardcoded paths:** Use `os.path.dirname(os.path.abspath(__file__))` for relative paths
3. **Standard library + allowed packages:** `numpy`, `scipy`, `sympy`, `mpmath`, `matplotlib` (no `pandas`)
4. **Command-line configurable:** Accept parameters via `sys.argv` or a config block at the top
5. **Reproducible:** Set random seeds explicitly: `np.random.seed(42)`
6. **Output to correct locations:** Images → `images/`, data → `data/`, stdout → captured in report
7. **Progress indication:** Use `print(..., flush=True)` or `tqdm` for long-running computations
8. **Graceful error handling:** Catch exceptions, print informative messages, exit with non-zero code on failure

### 3.2 Execution Workflow

For each module:

```
┌──────────────────────────────────────────────────────┐
│ 1. WRITE: LLM writes Python script(s) to src/        │
├──────────────────────────────────────────────────────┤
│ 2. EXECUTE: LLM requests UI to run the script         │
├──────────────────────────────────────────────────────┤
│ 3. CAPTURE: UI returns stdout, stderr, exit code      │
│            Generated images saved to images/          │
│            Generated data saved to data/              │
├──────────────────────────────────────────────────────┤
│ 4. DEBUG: If exit code ≠ 0 or output incorrect:       │
│          LLM analyzes error, modifies script,         │
│          re-executes (repeat until success)           │
├──────────────────────────────────────────────────────┤
│ 5. VALIDATE: Check output against validation criteria │
├──────────────────────────────────────────────────────┤
│ 6. REPORT: LLM composes module_XX_report.md           │
│           Embedding captured output and images        │
├──────────────────────────────────────────────────────┤
│ 7. COMMIT: Git commit with Conventional Commits msg   │
└──────────────────────────────────────────────────────┘
```

### 3.3 Error Handling Protocol

| Error Type | Response |
|:-----------|:---------|
| **Syntax error** | Fix immediately; re-execute |
| **Import error** | Check `requirements.txt`; install missing package |
| **Runtime error** | Analyze traceback; add debugging `print` statements; isolate failing component |
| **Numerical instability** | Increase precision (`mp.dps`), check convergence, add regularization |
| **Incorrect result** | Add validation checks; compare with hand-calculated or known results |
| **Timeout (>15 min)** | Save checkpoint automatically (all long-running scripts must handle SIGTERM/timeout gracefully). Verify partial output saved. Resume from checkpoint in next execution. If no checkpoint exists, reduce problem size. |
| **Out of memory** | Use generators; reduce array sizes; process in batches |

### 3.4 Performance Guidelines & 15-Minute Hard Cap

**HARD CONSTRAINT:** No single code execution may exceed **15 minutes (900,000 ms)**. The DeepChat execution environment enforces a timeout at 1,800,000 ms, but the project standard is stricter: every Python script must complete within 900 seconds. This constraint forces modular, checkpoint-able code design.

**Execution timeout parameter:** `exec(background=True, timeoutMs=900000)`

| Module | Expected Runtime | Strategy for 15-Minute Cap |
|:-------|:----------------|:---------------------------|
| Module 1 | < 10 seconds | Trivial — pure rational arithmetic. Single execution. |
| Module 2 | < 30 seconds | Series converge rapidly ($e^{-\beta p^{2n}}$). Single execution. |
| Module 3 | 1–5 minutes | Product over many primes; use vectorization. Single execution. |
| Module 4 | < 1 minute | Integer-only $\Gamma_p$; few kinematic points. Single execution. |
| Module 5 | < 1 minute | Simple 1D recursion. Single execution. |
| Module 6 | 30–90 minutes (total) | **MUST SPLIT.** Compute zeta zeros in **batches of 10,000** (~10 min each). Save partial results to `data/06_zeros_batch_N.json` after each batch. Merge in final step. Total: 10 batch executions + 1 merge. |
| Module 7 | < 1 minute | Enumeration over finite special points set. Single execution. |
| Module 8 | 60–180 minutes (total) | **MUST USE CHECKPOINTING.** The symbolic regression search must save its state to `data/08_checkpoint.json` every ~10 minutes. Resume from checkpoint in subsequent executions. Alternatively, split the pre-registered search space into independent partitions and run each as a separate sub-execution. |
| Module 9 | 30–90 minutes (total) | **MUST SPLIT.** Each null model runs as an independent execution (5 null models × ~5–10 min each). Merge results in a final report-assembly step. |
| Module 10 | < 5 minutes | Report assembly only. Single execution. |

### 3.5 Long-Running Module Protocol

For any module whose total compute exceeds 15 minutes, follow this protocol:

```
┌──────────────────────────────────────────────────────────┐
│ 1. DESIGN: Identify natural split points                  │
│    - Batch processing (Module 6: zeros in blocks of 10⁴)  │
│    - Independent sub-tasks (Module 9: one null per run)   │
│    - Periodic checkpointing (Module 8: save state + tree) │
├──────────────────────────────────────────────────────────┤
│ 2. EXECUTE: Run each chunk with timeoutMs=900000          │
│    Use background=True; poll with process(action='poll')  │
│    Save partial results to data/XX_chunk_N.json           │
├──────────────────────────────────────────────────────────┤
│ 3. VERIFY: After each chunk, validate partial output      │
│    Check file size, JSON validity, expected count         │
├──────────────────────────────────────────────────────────┤
│ 4. MERGE: Final execution combines all chunks             │
│    Produces the complete deliverable (data, images)       │
├──────────────────────────────────────────────────────────┤
│ 5. REPORT: Module report documents the split strategy     │
│    and confirms all chunks were executed successfully     │
└──────────────────────────────────────────────────────────┘
```

## 4. Testing Standards

### 4.1 Unit Tests

Every class and function in Modules 1–3 must have unit tests:

```python
# src/test_foundations.py
import pytest
from src.padic import PadicNumber

def test_padic_norm():
    """p-adic norm of 2 in Q_2 should be 1/2."""
    x = PadicNumber(2, 2)
    assert x.norm() == 0.5

def test_padic_product_formula():
    """|xy|_p = |x|_p |y|_p."""
    x = PadicNumber(2, 6)    # 6 = 2 * 3, |6|_2 = 1/2
    y = PadicNumber(2, 10)   # 10 = 2 * 5, |10|_2 = 1/2
    xy = x * y                # 60 = 2^2 * 3 * 5, |60|_2 = 1/4
    assert abs(xy.norm() - x.norm() * y.norm()) < 1e-15

def test_adele_product_formula():
    """Product formula for rational 12/25."""
    from src.adele import Adele
    from fractions import Fraction
    a = Adele.from_rational(Fraction(12, 25))
    assert a.norm_product() == Fraction(1, 1)
```

Run with: `python -m pytest src/test_foundations.py -v`

### 4.2 Integration Tests

After each phase, run a cross-module integration test:

